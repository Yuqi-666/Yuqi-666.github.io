
## 1.输入
接收用户输入的**单一目标**以及用户提供的**补充资料**，包含用户的背景信息，用户搜集的相关书籍，视频，可作为骨架参考或以”完成其阅览“作为关键成果。
## 2. 按层级拆解：
   - 层级定义：level=1（目标，o）、level=2（关键成果，kr）、level=3（任务,task）、level=4（子任务,subtask）。
   - 拆解规则：
     - level1（目标）：为用户输入的原始目标。
     - level2（KR）：将目标拆解为若干关键成果，需符合SMART原则（具体、可衡量、可实现、相关性、时限性），共同支撑目标达成。
     - level3（任务）：每个KR拆解为若干任务，为支撑KR达成的具体动作，estimated_time限制在1-4h（超出需将其子任务分割出去一起形成新的任务）。
     - level4（子任务）：每个任务可拆解为若干子任务（若任务无需拆分则可不设），为任务的细化执行步骤。
3. 节点信息规范：
   - 每个节点需包含4个字段：
     - id：唯一标识，格式为“level类型_父级序号_层级序号”（如task_3_1表示第三号关键成果的第一号任务）
     - level：1-4的数字，对应目标、KR、任务、子任务。
     - content：节点的具体内容（如目标内容、KR描述、任务、子任务动作）对于关键成果，内容要简洁（不能出现预估时间，如一周内），对于任务和子任务：在涉及用户给出的资料时要以其书名号（简洁起见，可使用代称，如课程A）+索引的方式引用（例如某子任务内容为：阅读《入门C++》视频第1.3小节 指针的使用方式）。
     - estimated_time：预估完成时间，格式为Hhmmin（如2h30min，0h30min），数值为子级节点estimated_time的总和。
## 3. 确定连接关系：
### 定义：
- **节点状态类型**：仅三种 —— 未进行、进行、完成。
- **直接依赖类型**：
    1. 直接顺序依赖：同 level 节点间，A 完成是 B 开始的前提，A 为 B 的前置节点。
    2. 直接完成依赖：父 - 子 level 节点间，A 为 B 的父节点，B 是完全依赖链末端，B 完成为 A 完成的前提。
- **节点连接规则**：仅体现两种直接依赖，连接方向为 “from 被依赖节点 id → 依赖节点 id”。
- **节点状态判定**：
    1. 开始进行：需所有指向它的同 level 节点（直接顺序依赖的前置节点）均完成。
    2. 完成：需所有指向它的子 level 节点（直接完成依赖的子节点）均完成。
- **依赖链定义**：由两种依赖连接形成的链条，任意截取的片段均为依赖链，末端节点完成即代表整条链所有节点完成。
- **完全依赖链定义**：完全依赖链的末端节点是其同level依赖链的末端（例如，将一个关键结果并入某条依赖链（该依赖链末端节点指向此关键结果）时，此关键结果虽为这条依赖链的末端节点，但因它还依赖其他关键成果，并非完整依赖链的末端节点，故不能指向目标。）
## 4. TSV格式输出：
   - 输出包含两个TSV表格，分别为“nodes”和“edges”(无需输出标题)，表格间用单独一行`******`分隔。
   - 每个表格首行为表头（字段名），后续行为数据行，字段间用制表符（\t）分隔，无多余空格或换行。
   - nodes表头：id\tlevel\tcontent\testimated_time
   - edges表头：from\tto
## 5. 提示：
1. 子任务严格遵循“1-3个关联内容”原则，禁止将多个不相关或过多数量的视频小节/章节合并为一个子任务，确保每个子任务执行时目标聚焦、可快速完成。
2. 若任务对应章节/视频的子任务拆分后总时长超出4h，需拆分任务（如任务1拆分为任务1-1、任务1-2），每个拆分后的任务仍需符合1-4h的时长要求。
3. 规划需结合用户提供资料与专业实践，若用户资料仅覆盖理论，需补充对应的实践类任务（如编写demo、完成小型功能模块），确保目标落地。
## 输出仅包含TSV完整内容，无需额外说明文字。
