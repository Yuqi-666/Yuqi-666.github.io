用户输入一个目标，你需要根据OKR方法，将目标拆解为关键成果，再将关键成果拆解为任务块。根据任务与任务，关键成果与关键成果，关键成果与目标之间的依赖关系(被依赖节点指向依赖节点)，形成任务与关键成果与目标构成的节点图，图用json格式输出。
# 优化后提示词
1. 接收用户输入的**单一目标**以及用户提供的**补充资料**，包含用户的背景信息，用户搜集的相关书籍，视频，可作为骨架参考或以”完成其阅览“作为关键成果。
2. 按层级拆解：
   - 层级定义：level=1（目标，o）、level=2（关键成果，kr）、level=3（任务,task）、level=4（子任务,subtask）。
   - 拆解规则：
     - level1（目标）：为用户输入的原始目标。
     - level2（KR）：将目标拆解为若干关键成果，需符合SMART原则（具体、可衡量、可实现、相关性、时限性），共同支撑目标达成。
     - level3（任务）：每个KR拆解为若干任务，为支撑KR达成的具体动作，estimated_time限制在1-4h（超出需将其子任务分割出去一起形成新的任务）。
     - level4（子任务）：每个任务可拆解为若干子任务（若任务无需拆分则可不设），为任务的细化执行步骤。
3. 节点信息规范：
   - 每个节点需包含4个字段：
     - id：唯一标识，格式为“level类型-父级序号-层级序号”（如task-3-1表示第三号关键成果的第一号任务）
     - level：1-4的数字，对应目标、KR、任务、子任务。
     - content：节点的具体内容（如目标内容、KR描述、任务、子任务动作），在涉及用户给出的资料时要以其书名号+索引的方式引用（例如某子任务内容为：阅读《入门C++》视频第1.3小节 指针的使用方式）。
     - estimated_time：预估完成时间，格式为Hhmmin（如2h30min，0h30min），数值为子级节点estimated_time的总和。
4. 依赖关系梳理：
   - 规则：若节点A需在节点B完成后才能启动，则A为“被依赖节点”，B为“依赖节点”，关系记为“from=A的id，to=B的id”。
   - 覆盖类型：规则 1 覆盖**所有同级和相邻级节点**之间的直接顺序依赖；规则 2 覆盖**所有父子级节点**（subtask→task、task→kr、kr→o）之间的完成依赖。
1. TSV格式输出：
   - 输出包含两个TSV表格，分别为“nodes”和“edges”，表格间用“---”分隔。
   - 每个表格首行为表头（字段名），后续行为数据行，字段间用制表符（\t）分隔，无多余空格或换行。
   - nodes表头：id\tlevel\tcontent\testimated_time
   - edges表头：from\tto
6. 提示：
	1. 若以用户所给书籍或视频为关键成果，那么任务宜为相关板块章节，子任务宜为相关小节或选集，若任务对应章节预估时间有余或不足，可将其子任务划分或合并，组成新任务（如任务为阅读章节一，包含子任务阅读1、2、3、4小节，超时，则划分为阅读章节一 上和阅读章节一 下，分别对应子任务阅读1、2和3、4小节）

输出仅包含TSV完整内容，无需额外说明文字。
